LTTS · Apr 2021 – Jun 2022
Healthcare Management Platform — Clinical Workflow & Patient Portal UI
APR 2021 — JUN 2022 · 1 YR 3 MOS · L&T TECHNOLOGY SERVICES

A healthcare management platform for hospitals and clinical facilities covering patient registration, appointment scheduling, clinical workflow management, and EHR-integrated record viewing for doctors, nurses, and administrative staff.

Problem
Doctors and nurses were navigating overly complex screens designed for software engineers, not clinical staff. High error rates during patient documentation and frequent escalations to IT during active consultations indicated a fundamental UX failure in the existing system.
What I designed
Spent two weeks doing task analysis with doctors and nurses before designing anything. Designed distinct role-based interfaces for clinical and administrative staff — doctors got a consultation-optimised view, nurses got a patient monitoring view, admin got a scheduling-first view. Outcome

🔬
LTTS · Nov 2020 – Mar 2021
Biomedical Equipment — Control Interface Design
NOV 2020 — MAR 2021 · 5 MOS · L&T TECHNOLOGY SERVICES

Control interface design for a specialised biomedical equipment system used in regulated clinical environments — designing the operating UI that clinicians and biomedical technicians use for device setup, calibration, real-time monitoring, and alert management.

Problem
The device had 40+ distinct hardware states, safety-critical alert conditions, and complex operating sequences. Poorly communicating any of these through the interface wasn't just a UX problem — in a clinical environment, it was a patient safety risk. The engineering team had deep hardware expertise but no framework for translating device states into a usable interface.
What I designed
Documented all 40+ device states before designing a single screen. Worked with the clinical advisor to create a three-tier alert hierarchy — immediate action, warning, and informational — each visually distinct through colour, icon weight, and screen position. Designed setup, calibration, monitoring, and alert management screens in Figma with full state-by-state annotations for the engineering team.
Outcome
The device state documentation I created became a reference used by the engineering team itself. The alert hierarchy design was validated by the clinical advisor as meeting safety-critical usability standards for regulated medical device software.

Alert & notification system — three-tier hierarchy
🔴
Critical — Immediate action required
Device shutdown sequence initiated. Patient Side Manipulator not responding. Verify connection and restart before proceeding.
🟡
Warning — Review before next action
Calibration drift detected on Instrument Arm 2. Recalibrate before next operating session to maintain accuracy thresholds.
🔵
Informational — No action needed
Session log saved successfully. Next scheduled maintenance window: 22 Jan 2026, 06:00 AM.

This three-tier alert hierarchy was designed for the biomedical equipment project at LTTS — where alert fatigue in a clinical environment was a genuine patient safety risk. Critical alerts take over the primary viewport. Warnings appear inline without blocking workflow. Informational alerts surface in a secondary panel. Visual differentiation uses colour, icon weight, and screen position — not just colour alone, to remain accessible for colour-blind users.

Alert Component — Usage Guidelines
When to use
Use Alert when the user needs to know about a system state that requires their attention. Do not use Alert for marketing messages or non-actionable information — use a Banner or Toast instead.
Props
// Alert component props
variant: "critical" | "warning" | "info"
title: string // required
message: string // optional
dismissible: boolean // default: false for critical
action: { label: string, onClick: fn }
Do's and don'ts
✓ Do✕ Don't
Use action verbs in titleUse "Error occurred" as title
Tell user what to do nextUse Critical for warnings
Make critical non-dismissibleStack more than 3 alerts
